## Patterns to Follow

### 1. Component Structure
- Use `'use client'` directive for client-side React components
- Wrap searchParams consumers in <Suspense> (see ResultsPage default export)
- Separate content from suspense boundary for cleaner code

### 2. State Management
- All filter/sort state is local React state (useState)
- No global state, no context providers
- localStorage for restore_token persistence only
- URL params read-only (profile_id, user_id, token)

### 3. Styling Conventions
- ALWAYS use Tailwind classes — NEVER inline style objects
- Design system colors:
  - Background: bg-[#F7F7F5]
  - Cards: bg-white rounded-2xl border border-gray-100 shadow-sm
  - Brand: indigo-600 (buttons, links, accents)
  - Text: gray-900 (headings), gray-600 (body), gray-400 (muted)
- Spacing: px-4, py-3, gap-2, space-y-4 (consistent rhythm)
- Interactive states: hover:, disabled:, transition-colors

### 4. API Calls
- /api/search endpoint:
  - Method: POST (NOT GET — critical for S-02)
  - Headers: Content-Type: application/json, X-Restore-Token: <token from localStorage>
  - Body: { profile_id: string }
  - Response: { data: { results, total, search_id }, meta: { threshold, max_score, searched_at } }
- Token validation: validateRestoreToken(profile_id, token) on server
- Error handling: 401 redirect to /restore, 404 show error, 500 generic error

### 5. Error Handling Pattern
```typescript
try {
  const response = await fetch('/api/search', { ... });
  if (!response.ok) throw new Error('Search failed');
  const data = await response.json();
  // process data
} catch {
  setError('User-friendly message');
} finally {
  setLoading(false);
}
```

## Story S-02 Integration Point (CRITICAL)
The "Edit Preferences" panel re-runs search WITHOUT persisting to DB.
- Current /api/search endpoint is POST-only
- Body must contain { profile_id: string }
- Additional preferences (location, workMode, employmentType, salaryMin) are NOT currently supported by /api/search
- MITIGATION: S-02 panel will call POST /api/search with ONLY profile_id (no extra params)
  → This means "re-run" is effectively a simple re-fetch, NOT a preference override
  → If user wants to change preferences permanently, they use /preferences page
  → Panel is for "what-if" exploration without persistence
